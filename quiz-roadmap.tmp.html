Квизы: дорожная карта до «контролируемого тестирования»

Ниже — идеи и грубый план, как эволюционировать текущий модуль (самопроверка в Markdown) в полноценный механизм тестирования с учётом времени, числа попыток и привязкой к пользователю/бэкенду.

1. Два режима работы плагина/компонента

РежимДля чегоИсточник вопросовХранение состояния
Self-check (текущий)Лекции/самопроверкаMarkdown, source (JSON/YAML)Pinia + localStorage
Assessment (будущий)Тестирование с ограничениямиAPI бэкенда (JSON)Pinia + сервер (через API)

Компонент переходит в Assessment-режим, если в options передано что-то вроде mode="assessment" + session-id / test-id.

2. Что нужно на фронтенде

  1. Поддержка API-источника

    • source=http(s)://... — забираем вопросы по REST (GET /tests/:id/questions).
    • Для защищённых тестов — аутентификация (использовать уже существующие механизмы auth в проекте).
    • Вопросы/ответы приходят уже перемешанные или с seed от бэкенда; фронт только отображает.

  2. Сессия теста

    • Бэкенд выдаёт sessionId, таймер (timestamp окончания), лимит попыток.
    • Pinia хранит локальный прогресс только для автосохранения между страницами до отправки.
    • Каждая попытка отправляет payload sessionId + answers[] + elapsed.

  3. Ограничение времени

    • Компонент получает expiresAt (ISO timestamp).
    • Таймер обратного отсчёта (отдельный composable) и автоматическое завершение при истечении.
    • UX: показать «Осталось ХХ:YY», «Время вышло — ответы отправлены автоматом».

  4. Ограничение попыток

    • В options приходит attemptsLeft.
    • После отправки обновляем фронтовую метку (можно хранить в Pinia + localStorage) до ответа бэка.
    • Если попыток нет — компонент в режим просмотра (без чекбоксов?).

  5. Отправка результатов

    • POST /tests/:id/attempts (payload: sessionId, ответы, время, устройство).
    • Бэкенд возвращает статус: success, timeout, already-submitted, no-attempts.
    • В случае успеха — либо показываем результат, либо редиректим на страницу «Спасибо».

  6. Модификация UI

    • Добавить шапку: таймер, «Потытка 2/3», кнопки «Сдать сейчас», «Сбросить черновик».
    • В режиме экзамена скрывать «Верно/Неверно» до отправки (либо только summary).
    • Настройки опций: showFeedback=false, lockAfterSubmit=true, allowDraft=true.

3. Изменения в плагине/компоненте

Плагин (Markdown)

  • Добавить возможность указывать mode=assessment, test-id=..., session=auto.
  • Для страниц лекций всё остаётся как есть; новые параметры просто прокидываются.

Компонент Quiz.vue

  1. Новый интерфейс входных данных

    • props.options.mode (practice/assessment).
    • options.sessionId, options.testId, options.expiresAt, options.attemptsLeft.
    • options.fetcher?: () => Promise<{ questions, sessionId, expiresAt }> — если нужно подгружать динамически.

  2. Состояния

    • status: idle | answering | submitting | submitted | locked | error.
    • Pinia: вместо одной общей коллекции — quizStore.sessions = { [sessionKey]: { answers, expiresAt } }.

  3. Таймер

    • composable useCountdown(expiresAt, onExpire) — внутри setInterval/RAF.
    • При onExpire вызываем submitAnswers({ reason: "timeout" }).

  4. Отправка ответов

    • async submitAnswers(reason = "user")
      • блокируем UI (status="submitting");
      • отправляем на /api/tests/:testId/attempts;
      • по ответу обновляем status, attemptsLeft, results.

  5. Хранение в Pinia

    • Не сохранять сами вопросы (восстанавливать через sessionId + API).
    • Сохранять только выборы (answers), sessionId, expiresAt.
    • Ограничить размер (по тестам), периодически очищать устаревшие (например, если expiresAt < now - 1d).

  6. Обработка ошибок

    • Если API недоступен или таймер истёк раньше — выводим сообщение, даём возможность повторить (если позволено).
    • При already-submitted — получить финальный статус (например, GET /tests/:id/attempts/:sessionId).

4. Что потребуется на бэкенде

(кратко для ориентира)

  1. API для получения теста

    • GET /tests/:id/start → { sessionId, questions, expiresAt, attemptsLeft }
    • Вопросы уже перемешаны, answers — без указания правильных флагов (чтобы фронт не знал ответы).
    • При необходимости — вернуть seed, если порядок должен быть воспроизводим без сохранения на фронте.

  2. API для сохранения/завершения

    • POST /tests/:id/attempts (в теле sessionId, ответы, время)
    • Проверяет лимит попыток, таймер, персонифицирует по userId.
    • Возвращает результат (score, passed, attemptsLeft).

  3. API для повторного открытия

    • GET /tests/:id/attempts/:sessionId — чтобы фронт мог восстановить статус (например, при перезагрузке страницы после отправки).

  4. Авторизация

    • Всё через токен пользователя, который уже используется на сайте (например, bearer-cookie).
    • Бэкенд сам решает, кто может начать тест.

5. Переиспользование существующего кода

  • Markdown-плагин + Quiz.vue остаются главным «рендерером» вопросов.
  • Сессии Pinia + localStorage уже есть — их нужно адаптировать (добавить sessionId, не хранить вопросы).
  • Таймеры, API-вызовы, обработка статусов — добавляем слоями, не меняя базовый интерфейс questions.

6. Мини-план внедрения тестового режима

  1. Шаг 1: внедрить source=API — научиться грузить вопросы по HTTP (без авторизации/таймера).
  2. Шаг 2: добавить sessionId и автосохранение в Pinia по ключу testId.
  3. Шаг 3: таймер + отправка на API (без ограничений попыток).
  4. Шаг 4: ограничение попыток, status + режим «только просмотр» после исчерпания.
  5. Шаг 5: косметика (шапка теста, кнопки, сообщения, отчёты для пользователя).

Так у нас останется лёгкий режим (Markdown/лекции) и расширенный (экзамены), использующие одну и ту же основу. Когда будем готовы, можно дробить на отдельные компоненты, но пока лучше сохранить единый стек. Если нужны уточнения по API или UX — пиши, можно проработать детальнее.
